## 1. 产品目标与阶段

- [x] **MVP 要验证什么**  
  以**打通全链路**为首要目标：Android / iOS 端（薄壳 + H5）→ 自建后端 → 云部署 → **应用上架** → 可观测与基本风控；在可运行闭环上**再**验证商业可行性（留存/付费/成本等可后置量化）。

- [x] **首版是否付费/会员**  
  **首版（MVP/上市第一版能力范围）不上线付费**；在数据模型中**预留** `订单/权益/订阅` 等相关表与扩展点，便于后续接入。

- [x] **市场**  
  首版面向**中国大陆**；后续可扩展**海外**（时区/合规/云与模型策略单独评估）。

---

## 2. 技术栈（后端）— 已选

- [x] **语言 + 框架**：`TypeScript` + **NestJS**
- [x] **数据库**：**PostgreSQL**（主库，开源、云托管按量付费；**不以 MongoDB 作主库**）
- [x] **缓存**：**Redis**，**MVP 即引入**（会话、限流、配额、热点缓存等）
- [x] **ORM**：**Prisma**（**已接受**）
- [x] **大模型（开发期）**：优先使用**火山引擎（豆包/方舟等）**已有 API Key 做联调；**允许**使用境外模型 API，**不强制**数据不出境（以法务与最终合同为准）。

---

## 3. 技术栈（客户端）— 已选

- [x] **MVP 形态**  
  **各端 `WebView` 薄壳 + 内嵌 `demo/web` 同源 H5**（或部署后的 HTTPS 地址），**最快**打通与后端的**鉴权、API、部署、商店**流程。

- [x] **MVP 之后正式原生**  
  **采用 Flutter 单工程双端**（`clients/flutter/`，待初始化），**不**采用 Kotlin Multiplatform 作为后续主路线（理由见 ADR-003）。

- [x] **系统版本**  
  目标覆盖 **Android / iOS 各近 2 个大版本**；具体 `minSdk` / `iOS min` 在工程初始化时写入 `tech-stack` 与客户端 README。

---

## 4. AI、合规与运营

- [x] **内容安全**  
  不做重审核平台；**MVP 仍须具备最小兜底**（敏感/自伤/违法等**规则 + 安全模板/拒答/降级**），与「完全不做」区分——满足应用商店与基本产品责任底线。

- [x] **管理后台**  
  **不进入 MVP**；与付费、运营强相关的后台放入**商用第一版**及以后。
