<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client -> `POST /v1/insight/submit` | Untrusted clients provide `rawQuestion`, `questions` snapshot, and selected `answers`. |
| Submit validation -> LLM prompt | Only validated snapshots may influence conclusion prompt generation. |
| Submit validation -> Prisma persistence | Only validated completed rounds may be stored as `InsightRound`. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-04-01 | Tampering | `InsightService.assertAnswerShape` | mitigate | Reject question snapshots unless they contain exactly the expected three dimensions in order, unique question IDs, valid option counts, unique per-question option IDs, and exactly one answer per question. |
| T-01-04-02 | Information Disclosure | Submit error body | mitigate | Use one stable 422 `INVALID_INSIGHT_ANSWER` error for all submit-shape failures; do not reveal which tamper path was detected. |
| T-01-04-03 | Denial of Service | Submit validation boundary | mitigate | Validate arrays and maps synchronously before user upsert, LLM calls, or persistence, so malformed snapshots do not consume external LLM resources. |
| T-01-04-04 | Repudiation | Submit rejection path | accept | Existing request-id logging from 01-03 records submit errors; this plan does not expand request-id sanitization because it is outside the verification gap. |
| T-01-04-05 | Spoofing | Client-provided question snapshot | mitigate | Do not trust the snapshot blindly; enforce structural and semantic consistency against the canonical `QUESTION_DIMENSIONS` contract while preserving the no-session design. |
</threat_model>
