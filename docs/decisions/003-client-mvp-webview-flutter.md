# ADR-003：客户端路线 —— MVP 用 WebView + H5，正式端用 Flutter

- **状态**：已接受
- **日期**：2026-04-24
- **决策者**：项目维护者

## 背景

- **7 月首版** 需要尽快打通：壳应用 → 后端 → 部署 → **应用商店** 等**全链路**。  
- 现成资产为 **`demo/web`（Next H5）**，在 **WebView 薄壳**中即可复用大部分 UI/流程。  
- MVP 之后需 **iOS + Android 一套** 实现**可维护的正式客户端**，在**不考虑个人学习成本**、仅从**产品与技术最匹配**出发，需在 **Flutter** 与 **Kotlin Multiplatform (KMP)** 间定案。

## 决策

1. **MVP / 首版商店能力**  
   采用**各端原生（或各端小工程）`WebView` 薄壳** + 内嵌 **H5**（`demo/web` 构建物或同构 URL），**优先**满足「上架、HTTPS、与后端真联调」的**时程与风险**目标。  

2. **MVP 之后正式双端**  
   使用 **Flutter** 建设 **单工程双端** 客户端（工程占位 `clients/flutter/`，**待建**），作为**主要** C 端形态演进方向。

3. **不采用** Kotlin Multiplatform 作为**与 Flutter 对等的、后续主路线**的默认选项。

## 为何是 Flutter 而非 KMP

| 维度 | Flutter | Kotlin Multiplatform |
|------|---------|------------------------|
| **单一代码的 UI 共享** | 一套 **Widget** 跑双端，**一致性强** | 常规模型为 **KMP 共享非 UI + iOS(Swift) / Android(Kotlin) 各写一套 UI**，**两条 UI 管线** 长期成本高；**Compose Multiplatform** 在 **iOS 侧** 成熟度与**生态/商店案例**（相对 Flutter）在消费级 App 上**风险更高**（截至本决策时点的普遍认知） |
| **商店/IAP/插件** | 支付、推送、WebView、**内购**、埋点、第三方 SDK 的**文档与现成方案**多 | 尤其 **iOS IAP**、**微信 SDK** 等，双端**分别对接** 概率高、周期难估 |
| **与当前 MVP** | 壳可短期保留，**H5 与 API** 可平滑**迁移**到 `WebView`/`in_app_webview` 再逐步**替换**为原生页 | 同理可嵌 H5，但**正式期**若坚持双原 UI，**不解决**「一团队维护两套 UI」的结构性问题 |
| **团队/招聘** | 与「跨端一致体验」的**产品目标**更一致 | 若组织未来 **100% Kotlin 后端 + Android 为主** 再评估 **KMP 做共享层** 更有优势 |

> 结论：从 **消费级双端、一套 UI、上架与三方集成** 出发，**Flutter 更适合**作为 FavorMe 的**正式**客户端主技术；**KMP 不排除** 今后作 **共享网络/数据层** 或 **子模块** 的**补充**，但**不**与 Flutter **并列** 作为默认双端主栈。

## 后果

- **正面**：MVP 极快；正式期 **一维护面**、**一致交互**、降低 **7 月后** 的迭代**分叉**。  
- **风险**：`WebView` 期有**性能/体验**与 **Cookie/存储** 边界，需在「薄壳 + H5」阶段**定清** 与后端的**鉴权域**、**H5 与 原生** 的**权限**（相机、相册、推送）**交接面**。  
- **待办**：`clients/android`、`clients/ios` 的薄壳**规范** + `clients/flutter` **新工程**创建策略（见 `clients/flutter/README.md`）；把 **min 系统版本** 定数字。
