# Phase 2 — UI Design Contract

> Visual and interaction contract for Android 对话与选题 UI. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

## Design System

| Property | Value |
|----------|-------|
| Tool | Manual Flutter design contract |
| Preset | not applicable |
| Component library | Flutter Material widgets with custom themed wrappers |
| Icon library | Material Icons only when needed; no microphone/voice icon in Phase 2 |
| Font | System CJK stack: PingFang SC / MiSans / Noto Sans CJK SC / system sans |

Source: `.planning/phases/02-android-ui/02-CONTEXT.md` locks Flutter-first implementation in `clients/flutter`, Android-only delivery with iOS-reusable structure, and a high-fidelity soft green/beige style based on the provided screenshots/Figma reference. No `components.json`, Flutter app, or existing Flutter token system is present, so this phase uses an explicit manual contract.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, progress label gaps, option indicator spacing |
| sm | 8px | Compact vertical gaps, card internal micro-spacing |
| md | 16px | Default element spacing, screen side padding, option gaps |
| lg | 24px | Card padding, input-to-content separation |
| xl | 32px | Major vertical breaks between question/result sections |
| 2xl | 48px | Empty-state breathing room and loading-card vertical centering |
| 3xl | 64px | Top safe-area to first content group on the initial screen |

Exceptions: touch targets must be at least 44px high even when their visual padding is smaller; bottom input capsule minimum height is 56px; circular send button is 44px.

## Typography

Use exactly these four size roles and two weights: regular `400` and semibold `600`.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Caption / progress | 12px | 400 | 1.4 |
| Body / option text | 14px | 400 | 1.5 |
| Action / card label | 16px | 600 | 1.35 |
| Heading / result title | 22px | 600 | 1.2 |

Rules:
- Question titles use `16px / 600 / 1.35`.
- Long result text uses `14px / 400 / 1.6` inside a scrollable card area if content exceeds the viewport.
- Buttons use `16px / 600`; secondary text buttons may use `14px / 400`.
- Do not introduce extra display sizes in Phase 2.

|------|-------|-------|
| Dominant (60%) | `#F6F5EF` | Screen background, safe-area fill, warm empty space |
| Secondary (30%) | `#FFFFFF` | Question cards, result card, input capsule, modal-like surfaces |
| Accent (10%) | `#6F9F72` | Send button, selected option pill/check, progress mark, retry action, “再问一次” affordance |
| Destructive | `#C65A4A` | Destructive actions only; Phase 2 has no destructive user action |

Accent reserved for: send button, selected option indicator, current progress marker, inline retry action, and “再问一次/回到首页” primary affordance. Do not use accent for every label or paragraph.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | 发送问题 |
| Empty state heading | 你今天想问什么？ |
| Empty state body | 输入一个正在纠结的选择，我会用三问帮你看见当下更偏向的 51%。 |
| Question input placeholder | 输入你纠结的问题… |
| Generating questions loading | 正在整理你的三问… |
| Generating conclusion loading | 正在生成倾向分析… |
| Question progress | 第 {current} / 3 问 |
| Back action | 上一题 |
| Retry action | 重试 |
| Result primary action | 再问一次 |
| Result secondary action | 回到首页 |
| Invalid question error state | 这个问题还不够明确。换个更具体的说法，我再帮你拆成三问。 |
| Network/server error state | 刚刚没有连上服务。请检查网络后重试。 |
| Destructive confirmation | none — Phase 2 has no destructive actions |

Copy rules:
- The UI must never imply medical diagnosis, fortune-telling certainty, or a command that decides for the user.
- Use “倾向”“更偏向”“当下” language for results; avoid “你必须”“注定”“诊断为”.
- Error cards explain the problem and the next action in one short sentence.

## Interaction Contract

### State Machine

The main page is a single Flutter screen with a finite state machine:

| State | Required UI |
|-------|-------------|
| `idle` | Soft empty homepage; the empty-state heading/body and bottom input capsule are the primary visual anchor; input enabled, no chat history list |
| `questionSubmitted` | User question bubble remains visible; input disabled while loading begins |
| `generatingQuestions` | Card-level loading with “正在整理你的三问…” and light breathing/typing motion |
| `answeringQuestion` | One question card at a time, progress `第 1 / 3 问`, options shown as large rounded pills |
| `submittingAnswers` | Existing selected answers remain visible; current card shows “正在生成倾向分析…” and actions disabled |
| `showingResult` | Result card with conclusion text and bottom actions “再问一次” and “回到首页” |
| `error` | Card-level error at the current failed step, retry scoped to that step |

### Question Flow

- Render exactly one question card at a time after the API returns the three-question snapshot.
- Store the full `raw_question`, `questions` snapshot, and selected answers in local state until submit completes.
- Selecting an option advances to the next card after a short 120-180ms pressed/selected feedback.
- On question 2 or 3, show `上一题`; returning preserves the previous selection and allows replacement.
- There is no submit overview page. Completing question 3 immediately submits all answers.
- A completed round ends by clearing current UI state and returning to `idle`; it must not imply persistent multi-round context.

### Input Bar

- Bottom input follows the screenshot: white rounded capsule, soft shadow, placeholder text, green circular send button.
- Remove the left microphone/voice slot entirely. Do not show a disabled microphone.
- Send is enabled only when trimmed input has meaningful text; disabled state uses `Text secondary` and no accent fill.
- During generation/submission, input and send are disabled to prevent duplicate calls.

### Cards And Surfaces

- Cards use white surfaces, 24px corner radius, optional soft border, and `Shadow soft`.
- On the idle screen, the empty-state prompt and bottom input capsule define the hierarchy; do not add extra hero artwork or secondary modules that compete with the input.
- Question cards should sit in the visual center/upper-middle of the available area, not as a dense form list.
- Result card can be taller and scroll internally for long conclusion text; keep bottom actions visible.
- Option pills use 999px radius, minimum 48px height, `16px` horizontal padding, and high-contrast selected state.

### Loading, Error, And Retry

- Questions generation failure retries `POST /v1/insight/questions` using the preserved input.
- Conclusion generation failure retries `POST /v1/insight/submit` using the preserved question snapshot and selected answers.
- Invalid input `422 INVALID_QUESTION_INPUT` returns to editable input state, preserves the user's original text, and shows the invalid-question error card.
- Network timeout and 5xx errors remain at the current flow position and show the retry action.
- Toasts may supplement diagnostics, but cannot be the only error surface.

### Motion

- Use light motion only: card fade-in plus 8px upward settle, button press scale to `0.98`, and subtle loading breathing/typing animation.
- Motion duration: 120-220ms for press/selection, 220-320ms for card entrance.
- Do not add heavy page transitions, complex character animation, or confetti.

### Accessibility

- All tappable controls must be at least 44px high.
- Text contrast must remain readable on `#F6F5EF` and white cards.
- Loading states expose semantic labels equivalent to the loading copy.
- The selected option must be conveyed by text/semantics, not color alone.
- Back navigation on Android system back from question 2/3 returns to the previous question; from question 1 returns to editable input with original text preserved.
